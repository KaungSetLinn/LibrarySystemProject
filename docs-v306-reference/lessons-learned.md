# 教訓（Lessons Learned）v3.0

> 作成者: Y.Toyoda / 更新日: 2026-05-04

## LL-01 SSOT を最初から明文化する
- **事象**: `library-system-config.txt` がコード上で読まれない（BUG-01）。
- **根本原因**: 設定の SSOT がどれかが文書・コードどちらにも明示されていなかった。
- **教訓**: 設定優先順位は実装着手前に文書化し、コード冒頭にもコメントとして残す。

## LL-02 状態変更系メソッドは正規化関数を経由する（BUG-V2.0）
- **事象**: 通知既読化で is_read の型が 0/1/true/false で揺れる。
- **教訓**: 状態変更後は必ず正規化関数を通し、再取得した値を返す。回帰テスト常設。

## LL-03 採番器は MAX(id) を起動時に必ず同期（BUG-V2.1-01）
- **事象**: localStorage クリア後の採番が 1 から始まり、既存IDと衝突。
- **教訓**: `_nextSeq` の冒頭で必ず `_scanMaxId` を実行する。`max(scan, counter)+1` を採用。

## LL-04 5仕様書とコードのバージョンは同時更新する
- **事象**: v2.x で要求 v2.1 / 外部 v2.2 / 内部 v2.1 / DB v2.2 / テスト v2.1 と不揃いに。
- **教訓**: 5仕様書とコードは同一バージョン番号で同時更新する。v3.0 で全て統一。

## LL-05 用語は最初から SSOT 化する
- **事象**: 「ユーザー」「利用者」「学生」「会員」が混在。
- **教訓**: 共通用語集を最初に作り、全ファイルから参照させる。

## LL-06 localStorage は SSOT にしない
- **事象**: テスト用切替が永続化され、本番設定を上書きするバグ（BUG-02）。
- **教訓**: localStorage は「一時上書き / デバッグ専用」とし、SSOT は txt ファイルに置く。

## LL-07 仕様の核心定義は独立章に置く
- **事象**: 四層構造定義が §8.2.1 ハイブリッド構成節の対比表に依存（カゥン氏指摘）。
- **教訓**: 仕様の核心は独立章で定義し、他章はそれを参照する形にする。

## LL-08 全関数に JSDoc + 仕様書トレーサビリティを残す
- **事象**: v2.x で関数の意図がコメントから読み取れず、レビュー時に多大な時間を要した。
- **教訓**: `@param @returns @throws @spec @example` の5タグ必須化。

## LL-09 `form.reset()` 使用時の必須セット（v3.0.1 新設）
- **事象**: BUG-19～BUG-23 が連続発生。`form.reset()` が JS で動的に埋めた値
  （ブラウザ名、`<select>` の現在値、設定サマリ等）を一括で消し、
  再描画を呼ばない限り二度と表示されない。
- **根本原因**: `form.reset()` は `<input value>` の HTML 既定値に戻す仕様。
  JS で動的に埋めた値は HTML 既定値とは別主体のため、一括で消える。
- **教訓**: `form.reset()` を呼ぶ篇所では、以下を4点セットでペア実行する：
  （1）JS で動的に埋めた input の再描画関数を呼ぶ
  （2）`<select>` の初期表示値を明示的に再代入
  （3）設定サマリ等の `<span>` も再描画
  （4）クリア完了を aria-live 経由で a11y 通知
- **詳細**: `docs/coding-rules.md` LL-09 参照。
- **適用済み修正**: BUG-19～BUG-23（5件）v3.0.1 で完全反映。
